# Curie Competitive Teardown — Product & Engineering/Architecture

Scope: technical analysis of eight products relevant to Curie's "self-writing lab memory" problem — collision detection against prior work, using Slack as substrate, semantic + keyword retrieval, and belief tracking over time. Marketing language is stripped out wherever possible in favor of data model, retrieval architecture, and platform hooks.

---

## 1. Bubble Lab / @Pearl

**Core problem & UX loop.** Bubble Lab is the closest structural analog to Curie: same dual-name pattern (product "Bubble Lab," agent handle "@Pearl"), same "operate inside Slack" thesis. Pearl is an ops copilot — a user describes an automation in natural language in Slack, Pearl maps the intent onto available integration primitives, and the result is a runnable workflow, not just a chat answer. The loop is: user prompt → Pearl proposes/builds a workflow → workflow executes against connected tools (Notion, Jira, HubSpot, Stripe, Google Workspace) → result posted back in Slack. [Automate Ops in Slack with Pearl by Bubble Lab](https://www.linkedin.com/posts/y-combinator_bubble-lab-yc-w26-supercharges-your-ops-activity-7432107080788099072-sOs5)

**Memory / data model.** There is no knowledge graph or vector store as the primary artifact — the "memory" *is* the compiled workflow. BubbleLab (the open-core engine, Apache-2.0, TypeScript, 1.1k GitHub stars) is a **graph-based execution runtime**: nodes called "Bubbles" are typed units of computation (integration trigger, transform, conditional branch, LLM call, API action) chained into a DAG. A user's natural-language request gets compiled by an LLM into a `BubbleFlow` — literal TypeScript classes with a `handle(payload)` method that instantiate and `.action()` chain Bubbles:

```typescript
export class RedditNewsFlow extends BubbleFlow<'webhook/http'> {
  async handle(payload: RedditNewsPayload) {
    const scrapeResult = await new RedditScrapeTool({ subreddit, sort: 'hot', limit }).action();
    const summaryResult = await new AIAgentBubble({
      message: `Analyze these top ${posts.length} posts...`,
      model: { model: 'google/gemini-2.5-flash' },
    }).action();
    return { subreddit, postsScraped: posts.length, summary: summaryResult.data?.response, status: 'success' };
  }
}
```
This is genuinely interesting: the "knowledge" that persists between runs is **executable code**, versioned and inspectable, not embeddings. Execution is observable — every run logs bubbles executed, token usage, memory peak, and duration. [GitHub — bubblelabai/BubbleLab](https://github.com/bubblelabai/BubbleLab)

**Retrieval architecture.** Not retrieval-first. Pearl's "search" is really an LLM doing intent→capability mapping against a fixed catalog of Bubble primitives (an agentic plan-then-execute pattern, but planning over *tools*, not over a document corpus). There's no RAG layer described in public materials — provenance is implicit in "which Bubble ran," not "which document was cited."

**Surfaces/events hooked.** Slack-native: @Pearl mentions trigger workflow creation/execution; the studio is also usable outside Slack (`app.bubblelab.ai`) for building/hosting flows independently.

**Steal:** the **DAG-of-typed-nodes execution trace as the audit log**. For Curie, every verdict (Clear/Collision) should be reconstructable as an ordered trace of tool calls (RTS query → List lookup → LLM verdict) with token/timing stats — this is what makes "we already tried this" defensible when someone pushes back, and it's cheap to build since your pipeline is already staged (preflight → RCS → verdict).

**Avoid:** Pearl's memory model has no first-class "knowledge base to check against" — it's optimized for *doing*, not *remembering and warning*. If Curie copies this pattern too literally, it will drift toward "build me a workflow" instead of "tell me if this collides." Keep the record-check loop (Slack Lists as source of truth) as Curie's center of gravity, not a Bubble-style capability catalog.

## 2. Glean

**Core problem & UX loop.** Company-wide enterprise search/answer engine across 250+ connectors. User asks a question (search bar, Assistant chat, or an agent built on Glean's Agent Builder) → Glean returns a synthesized, cited answer plus a system of related documents/people/activity. Newer "Agentic Engine 2" (Sept 2025) adds adaptive planning and parallel sub-agent orchestration for multi-step tasks. [Accelerating the journey to the superintelligent enterprise](https://www.glean.com/blog/live-fall-25-main)

**Memory / data model — this is the most relevant piece for Curie.** Glean explicitly builds a **knowledge graph** distinct from and layered *on top of* its search index. Core structure is the classic RDF-like **triplet**: `(subject, predicate, object)` — e.g., `(Engineer A, owns, Jira Ticket B)`, `(Document X, references, Project Y)`. Edges carry **properties**: timestamps, access-control, confidence scores, provenance. The graph is built algorithmically (not manually curated, because enterprise data is private and can't be eyeballed at scale) in five phases: (1) automated noun-phrase extraction, (2) frequency/prominence filtering to drop noise terms, (3) evidence-of-importance scoring (title presence, cross-linking, access frequency), (4) predicate identification (e.g., "document authored by," "project led by") prioritizing repeatable evidence, (5) continuous algorithmic refinement with no human in the loop. On top of the graph, Glean recently added a **personal graph** — a second graph that clusters atomic user activity ("created doc X," "attended meeting Y") into subtasks → tasks → themes → OKRs, built specifically because LLMs alone can't do multi-hop, deterministic enumeration ("list all account executives in Asia") reliably — they "overweight the closeness of terms" and conflate similarly-named entities (their own example: Claude 3.7 Sonnet vs 3.5 Sonnet v2 get merged by an LLM without a KG backstop). [How knowledge graphs work and why they are the key to context for enterprise AI](https://www.glean.com/blog/knowledge-graph-agentic-engine)

**Retrieval architecture.** Hybrid: a real-time crawler continuously ingests content/metadata feeding both **lexical and semantic (embedding) search**, plus the knowledge graph for structured/deterministic queries. The KG is specifically positioned as the fix for retrieval-augmented generation's known failure mode: LLMs are reliable for "simple fact recall... sometimes 100%... especially when the answer is explicit and nearby," but reliability drops sharply for multi-hop or indirect questions — this is precisely Curie's use case ("does this collide with three-months-ago's failed assay run, referenced obliquely in a thread that never used the word 'collision'"). Citations/provenance: fine-grained ACLs are baked into every graph edge so results are pre-filtered by what the querying user can already see in the source system — not a post-hoc filter.

**Surfaces.** Web app, browser extension, Slack/Teams/etc. connectors, MCP Gateway, Agent Builder for building custom agents on top of the graph+search substrate.

**Steal:** the explicit acknowledgment that **LLM-only retrieval degrades on multi-hop, deterministic, or enumerate-type queries** — and the fix (a structured, queryable layer beneath the LLM) rather than trying to prompt-engineer around it. For Curie, "has this exact combination of reagent + cell line + assay type been tried in the last 6 months" is exactly this kind of query — it should hit structured List data (deterministic), not rely on an LLM eyeballing search snippets.

**Avoid:** the five-year, always-on crawler-plus-graph investment. Glean's graph requires continuous algorithmic entity extraction/refinement at massive scale to stay non-stale; that's a multi-year infra bet, not a hackathon-deadline one. Curie should NOT attempt to build a general-purpose knowledge graph — it should treat Slack Lists as the pre-built structured graph substitute (rows/columns instead of triples) and spend zero engineering budget on generic entity extraction.

## 3. Dust

**Core problem & UX loop.** Horizontal enterprise agent platform — agents that "live in your stack" (Slack, Notion, GitHub, Confluence, Salesforce) and act on events, not just answer questions in a chat window. E.g., an agent comments on a PR based on the design doc that motivated it, or files a Jira ticket from a Slack bug report, auto-categorized to match internal conventions. Users `@dust` mention in Slack or DM; teams can bind a named agent (e.g. `@hr_help`) to a specific channel so users don't need to remember an agent's exact name. [Dust in Slack](https://docs.dust.tt/docs/slack)

**Memory / data model — architecturally the deepest match for Curie's "real-time hooks" need.** Dust runs entirely on **Temporal** as its orchestration/durable-execution backbone. Every external event (Slack message posted, Notion doc edited, GitHub PR opened) triggers a Temporal **Workflow**. That workflow does ingestion → enrichment → indexing, potentially spanning seconds to hours. Expensive steps (chunking, embedding) run as async Temporal **Activities** so they don't block the response path. At the center sits a **connectors service** managing real-time ingestion per source; Temporal Cloud handles retries, state, and horizontal worker scaling (10M+ Activities/day, "scaling just means running more Workers" — no custom infra). Everything else is Postgres + GCS + Kubernetes, instrumented with Datadog. [How Dust Builds Agentic AI with Temporal Workflows](https://temporal.io/blog/how-dust-builds-agentic-ai-temporal)

A second product, **Tracker**, is the direct conceptual ancestor of Curie's "belief updates" feature: it polls for **stale documents** based on internal activity signals and triggers agent-led updates to coding guidelines/internal process docs — described by Dust's CTO as "a kind of agent-for-agents" that "closes the loop between learning and doing." This is architecturally: poll → detect staleness signal → propose structured diff → (implicitly) human approval → write-back, all running as a retryable, resumable Temporal workflow (critical because staleness detection + LLM proposal + write-back is exactly the kind of multi-step, partially-failing pipeline Temporal is built for).

**Retrieval architecture.** RAG-agentic: Dust frames retrieval as iterative, not single-shot — "it pulls information, checks whether that information actually answers the question, and adjusts its strategy if the first attempt falls short" rather than one embedding search and done. [RAG Agent: What it is, How it works, and Why it matters](https://dust.tt/blog/rag-agent) Multi-source: agents retrieve from Notion/Slack/Drive/Salesforce/GitHub through one-click connectors and the platform handles cross-source permission enforcement and synthesis.

**Surfaces.** Slack (channel mention + DM + channel-bound named agents), GitHub, Notion, Confluence — all as both *read* sources (context) and *write* targets (actions: PR comments, ticket creation).

**Steal:** the **event-triggered durable workflow model for staleness/belief-tracking**. Curie's "hypothesis tracking + belief updates" feature is structurally identical to Dust's Tracker — this validates that pattern as sound, not speculative. Even without adopting Temporal itself (probably overkill for a hackathon), Curie's `pipeline/` (preflight, logging, ledger) should be designed as resumable, idempotent steps with explicit retry semantics, because "detect a new experiment references a stale hypothesis → propose an update → post to the ledger" is a multi-step pipeline exactly like Tracker's.

**Avoid:** full Temporal-as-infrastructure for a hackathon deadline (Jul 13 2026) — that's real production-scale plumbing (10M+ activities/day) solving a scaling problem Curie doesn't have yet. Also avoid Dust's "horizontal, connect-everything" scope creep; Curie's value is depth on one narrow workflow (experiment-plan collision), not breadth across Salesforce/GitHub/Confluence.

## 4. PromptQL (Hasura)

**Core problem & UX loop.** "Why is my AI analyst confidently wrong?" — LLMs answering business questions ("Is our WIP inventory at Seven Oaks trending low?") without knowing what "WIP," "low," or "Seven Oaks" mean in *this* company's vocabulary. User asks a natural-language question → PromptQL generates a multi-step **query plan in a DSL** that executes deterministically outside the LLM → results are computed by a real query engine (SQL/GraphQL/API), not hallucinated by the model. [The ultimate guide to semantic layers for AI](https://promptql.io/blog/the-ultimate-guide-to-semantic-layers-for-ai)

**Memory / data model.** A **"semantic supergraph"**: metadata-driven, YAML-encoded definitions of schema, relationships, business rules, auth rules, and constraints, auto-bootstrapped from structured (DB schemas) and unstructured (wikis, BI tool definitions) sources, fully version-controlled (SDLC-friendly — diffable, revertible). Distinct from a data warehouse: it's a **layer of meaning on top of** existing databases/APIs, not a copy of the data. Critically, it's designed to be a **living system**: user corrections/clarifications during usage feed back into the metadata model automatically ("closed-loop feedback cycle — query outcomes feed back into the metadata to improve accuracy over time"), with version control and automated regression checks so definition changes don't silently break existing answers.

**Retrieval architecture.** Not retrieval in the RAG sense at all — it's **query-plan generation + deterministic execution**. The LLM's job is narrowly scoped to translating "active users last fiscal quarter" into the *correct filters, joins, and aggregations*, which then execute via SQL/GraphQL/API against the real backend with **query pushdown** (filtering happens at the data source, not client-side) and access control enforced at query time, not after. This is architecturally the strongest "agentic plan-then-execute" pattern of the eight — the LLM plans, a non-LLM engine executes, and results are exact rather than approximately-retrieved.

**Surfaces.** SDK/app layer ("Multiplayer" cowork app), `@Tag` mention pattern (interesting: PromptQL also uses an `@` handle convention — `@Tag` — similar to Curie's `@Prior`), docs-driven API, no direct Slack integration surfaced in public docs as of this research.

**Steal:** the **"semantic layer as living, versioned, feedback-tuned metadata"** framing, and specifically the discipline of separating "what does this term mean in our business" (deterministic, versioned, human-auditable) from "how do I answer this specific question" (LLM's job, scoped narrowly). For Curie, this maps directly onto keeping the **Slack List schema and alias-expansion vocabulary as versioned, inspectable config** (not baked into a prompt) — when a scientist writes "the KO line" and the lab means a specific named cell line, that's a semantic-layer mapping problem, not something to re-derive from scratch in every LLM call. Also worth stealing: **query pushdown / execute-outside-the-LLM for anything deterministic** — Curie's collision check against Slack Lists should be a real filtered query (List filters), not "paste the whole list into the prompt and ask the LLM to eyeball it."

**Avoid:** building a general-purpose semantic-layer DSL. That's Hasura's decade-long bet on metadata-driven APIs; Curie doesn't need a query-plan compiler, it needs alias-expanded OR-queries against RTS and structured List lookups — a much shallower, more literal translation layer is sufficient and appropriate for the deadline.

## 5. Unblocked

**Core problem & UX loop.** Positioned explicitly against Glean: "search returns documents, a context engine returns answers." Problem: engineering knowledge is fragmented across code, PRs, Slack threads, and Jira tickets, and **conflicts between sources are the actual failure mode** — e.g., a Confluence page says JWT expiry is 15 minutes, the code sets it to 60 minutes, a Slack thread from last quarter explains why it changed and the docs were never updated. User asks a question (via Slack, IDE, CLI, MCP, or web) → Unblocked reads across all relevant sources → returns **one synthesized, citation-backed answer that has already resolved the conflict**, not a ranked list of possibly-contradictory documents. [Unblocked vs Glean: The Context Engine for Engineering Teams](https://getunblocked.com/blog/unblocked-vs-glean/)

**Memory / data model.** A **unified knowledge graph specifically for engineering sources** (25+ connectors: GitHub, GitLab, Bitbucket, Jira, Linear, Confluence, Notion, Slack, code) that explicitly performs **conflict resolution and reconciliation** as a first-class step, ranking by **authority, freshness, and source type** rather than textual relevance alone: "code outranks stale docs. Merged PRs outrank open drafts. Recent Slack decisions outrank year-old wiki pages." This ranking logic — not connector breadth — is presented as the core IP.

**Retrieval architecture.** Cross-source reasoning as a named capability: reads the merged PR + the Jira ticket that requested it + the Slack thread debating the approach + the code itself, and returns one synthesized answer with per-source citations — explicitly framed as "no tab switching, no manual cross-referencing." **Real-time, per-query permission enforcement** — not periodic ACL sync, which the company argues creates a security window where a search tool can surface something a source system has already revoked access to. Delivered to AI coding agents via **MCP-native** integration so agents get "pre-synthesized, conflict-resolved answers" instead of raw documents they'd have to reason over themselves (claimed 48% fewer tokens, 83% faster task completion on internal benchmarks vs. raw-document retrieval).

**Surfaces.** Web, macOS app, Slack, Microsoft Teams, MCP server, CLI, API — explicitly multi-surface with the same synthesized-answer contract everywhere.

**Steal:** **freshness/authority-based ranking as an explicit, named ranking function**, not an emergent property of embedding similarity. This is close to Curie's actual differentiator: "we already tried this and it failed" requires knowing that a failed-experiment record in a Slack List (structured, author-attributed, dated) should outrank an ambiguous Slack message that merely *mentions* a similar reagent in passing. Curie's ranking should explicitly encode: **List record (highest authority) > recent RTS message with clear outcome language > older RTS message > literature** — spelled out as a real tie-break rule, not left to LLM judgment. Also steal the **"return one synthesized verdict with citations, not a ranked list"** framing — this is literally Curie's Clear/Collision verdict UX.

**Avoid:** Unblocked's engineering-specific ontology (PRs, commits, git blame) doesn't transfer — don't try to force-fit "code" concepts onto lab science. Also be skeptical of the specific benchmark numbers (48%/83%) — they're self-reported marketing claims from a comparison blog post, not independently verified; cite them only as "a synthesized-answer architecture reduces agent token usage vs. raw retrieval," not as validated figures.

## 6. Guru

**Core problem & UX loop.** "Even AI can get it wrong" — the problem is trusting *any* knowledge source, human-written or AI-generated, without a verification mechanism. User searches or asks a question → gets an answer sourced from **Cards** → each Card visibly carries a verification status (verified/unverified), a timestamp, and a named verifier, so the reader can judge trust without asking anyone. [Keep content accurate with Guru's SME verification workflow](https://www.getguru.com/features/verification)

**Memory / data model.** **Card-based**, not graph-based: atomic units of knowledge (single topic/policy/procedure) each with metadata: assigned verifier (person or group, AI-suggested based on who created/last-edited the content), verification interval (weekly/monthly/quarterly/yearly/custom date, criticality-based — e.g., API policy cards reviewed monthly, team-directory cards yearly), and a trust score. This is a **human-in-the-loop staleness system**: when a Card nears its review deadline, Guru notifies the assigned verifier via Slack/email/in-app; they either **one-click re-verify** (still accurate) or **update-then-verify** (content changed). Cards that go unverified past deadline are auto-queued for archive review. Most importantly: **"Verified RAG"** — if a Card expires without re-verification, **the AI agent is restricted from using it** as a retrieval source. Verification status is a hard gate on the retrieval pipeline, not just a UI badge.

**Retrieval architecture.** RAG grounded exclusively in trusted content — the verification gate happens *before* retrieval, filtering the candidate pool to only currently-verified Cards, rather than retrieving broadly and hoping ranking surfaces the accurate version. Permission model layered on top: only users with Author/Workspace Owner/Admin roles can perform verification actions.

**Surfaces.** Slack (notifications), email, in-app, Card Manager (bulk filter/action by verification status), "My Tasks" dashboard (relevancy-sorted verification queue), Knowledge Agents (2026 addition — AI agents can now *auto-verify or auto-unverify* content, not just retrieve it).

**Steal:** **verification-status-as-retrieval-gate**. This is directly applicable to Curie's calibration mandate ("false collision is the unforgivable bug"): a stale or ambiguous prior-experiment record in the Slack List should be able to be marked (by a human) as unreliable/superseded, and Curie's collision logic should *exclude* or heavily downweight records flagged that way, rather than trusting every List row equally forever. Also steal the **one-click re-verify vs. update-then-verify UX split** — cheap for the reviewer when nothing changed, structured when it did. This maps well onto Curie's "belief update" posts: when Prior flags a hypothesis as possibly stale, offer the scientist a one-click "still holds" vs. "update it" choice rather than requiring free-form response.

**Avoid:** Guru's model assumes a dedicated content-maintenance workflow with named owners and scheduled reviews — that's a governance overhead lab scientists posting experiment plans in Slack will not tolerate as a blocking gate. Curie should treat verification as an *optional signal that improves ranking*, never as a hard requirement before a Slack List record counts as prior art — otherwise the collision-check becomes useless on day one when nothing has been verified yet.

## 7. Notion AI

**Core problem & UX loop.** Workspace-embedded AI: "Ask AI" queries across the user's entire Notion workspace (pages, databases, linked docs) and returns a synthesized, grounded answer rather than requiring manual search. Also generation/editing (drafting, rewriting) within the same document surface.

**Memory / data model — the deepest infra story of the eight, worth understanding in detail because it shows the *cost curve* of taking vector search seriously at scale.** Notion's public engineering retrospective ("Two Years of Vector Search at Notion") documents five major architecture bets in order:
1. **Nov 2023**: pod-based vector clusters (storage+compute bundled, sharded by workspace ID) hit capacity within a month; fixed by routing new workspaces to fresh "generation ID" clusters rather than live resharding — introduced two years of generation-routing complexity as technical debt.
2. **May 2024**: pod → **serverless** (storage/compute decoupled) — costs down 50% immediately, generation-routing complexity dissolved because capacity was no longer a pre-planned finite resource.
3. **Late 2024–early 2025**: serverless provider → **turbopuffer** — required a full re-index (proprietary storage format), used as an opportunity to also upgrade the embedding model simultaneously.
4. **Mid-2025**: **Page State** — every text span hashed with **xxHash (64-bit)**, stored in **DynamoDB**; on page update, the system compares hashes *before* deciding whether to re-embed. Metadata-only changes patch the vector DB directly without recomputing the vector — a meaningful cost optimization (avoid re-embedding on every trivial edit).
5. Shortly after: embedding generation moved from **Spark + external API** to **self-hosted models on Ray via Anyscale** — CPU preprocessing and GPU inference now run in a single pipeline, eliminating an S3 handoff between systems that previously existed as separate stages.

Net result: costs down ~90% from peak over two years. [Notion Vector Search Architecture: What Comes Next](https://zilliz.com/blog/notion-vector-search-next-problem) An independent analysis (Zilliz) frames this as Notion re-deriving, in two years, patterns Big Data took 15 to converge on: storage-compute separation, a Lambda-architecture split (Spark offline batch + Kafka real-time consumers, synchronized via Page State's hash-diffing), and a still-missing "declarative interface for vector operations" (no SQL-equivalent for "deduplicate this collection" or "backfill embeddings with a new model" — those remain multi-month systems-engineering projects, not routine operations). The same analysis flags Notion's *next* problem as **offline context engineering**: precomputed knowledge graphs of workspace semantic topology, and continuously-compressed user memory (old conversations need denser re-representation over time, not infinite raw retention).

Separately, Notion's native data model itself has a lightweight relational layer: database "relation" properties create bidirectional links (a Projects DB row related to People DB rows), **@mentions** create implicit backlink graphs surfaced per-page, and "rollups" compute live aggregates across relations. This is Notion's low-tech answer to "knowledge graph" — genuinely closer to what Slack Lists + `parent_item_id` hierarchy offers than to Glean's algorithmic triple-extraction.

**Retrieval architecture.** RAG: retrieves relevant blocks from the workspace to ground the model before generation, using the turbopuffer-backed vector index described above; routed through an internal "AI Gateway" that picks the model per task (reasoning/coding vs. creative generation vs. smaller specialized models for speed).

**Steal:** the **hash-before-embed pattern** (Page State) — don't re-embed/re-check content that hasn't semantically changed. For Curie, this maps to: don't re-run the full RTS+literature+List collision check if an experiment plan message is edited but the substantive content (reagents, method, hypothesis) is unchanged — hash the normalized plan text and skip re-verdict on no-op edits, saving RTS call budget (which is capped at ≤3 calls/verdict per the platform constraints already in Curie's spec).

**Avoid:** the entire vector-infrastructure buildout. Notion needed two years and five re-architectures to make embedding-based search cheap at hundreds of millions of workspaces; Curie has zero of that scale and zero of that timeline. Curie should not stand up its own vector database at all — lean entirely on Slack's own semantic search (RTS API, `disable_semantic_search: false` by default) and keep literature search to a narrow, cached, low-volume call pattern.

## 8. Slack AI / Slack's own agent platform (2025–2026)

This is not a competitor so much as the substrate Curie is built on — but it is the single most load-bearing piece of research here, because Curie's CLAUDE.md already encodes several platform facts as settled, and this research both confirms and sharpens them.

**Core problem Slack is solving.** Turning Slack from a place agents *post into* to a place agents can **query live, structured, permission-scoped workspace data from** without those agents having to build and maintain their own index/replica of Slack content (which would violate Slack's terms of service around retaining customer data).

**Memory / data model.** Slack itself does not expose a general knowledge graph to third-party apps. It exposes three *distinct* structured-data primitives apps can build on:
- **Real-time Search (RTS) API** (`assistant.search.context`) — live, permission-scoped search across messages/files/channels/canvases/users, **not an index a third party owns or persists** — this is explicitly why the spec says "never persist retrieved Slack data." [assistant.search.context method](https://docs.slack.dev/reference/methods/assistant.search.context)
- **Lists API** (`slackLists.*`, GA Sept 2025) — genuine structured tables inside Slack: `slackLists.create`, `.update`, `.items.create/.update/.delete/.deleteMultiple/.list/.info`, `.access.set/.delete`, plus bulk `.download.start/.get`. This is Slack's answer to "give an app a real database surface natively in the client," and it supports hierarchy via `parent_item_id`. [Introducing the Lists API](https://docs.slack.dev/changelog/2025/09/02/list-api)
- **Conversations API** (`conversations.history`/`.replies`) — for pulling a specific thread's full context once RTS has identified it as relevant, explicitly positioned as a "drill in" step *after* RTS narrows the search space, not a bulk-crawl mechanism.

**Retrieval architecture — directly confirms Curie's CLAUDE.md.** `assistant.search.context` requires an `action_token` from the triggering event payload when called with a **bot** token (present only on `app_mention` and `message.im` events) — a **user** token needs none. This is a deliberate scoping mechanism: it ties every RTS call back to a specific user-initiated event, preventing apps from running unbounded background searches. Phrasing the `query` param as a natural-language question triggers **semantic search**; `disable_semantic_search: true` forces pure keyword mode. Slack's own official guidance for context management (published as a first-party best-practices doc, not just an API reference) explicitly recommends: **structured state objects** (`{ goal, constraints, decisions, artifacts, sources }`) instead of re-injecting full thread history every turn; **progressive summarization** of older material; **token budgets per request**; and **drift detection** (has the current work diverged from the original goal — confirm with the user before big actions). This state-object shape is *exactly* the pattern Curie should adopt for its own `pipeline/` state, and Slack is explicitly recommending it, not just Curie inferring it. [Context management](https://docs.slack.dev/ai/agent-context-management)

Search filters support `term_clauses` (conjunctive-normal-form keyword matching) and Slack-search-bar modifiers (`before:`, `after:`, `has:`) embedded directly in the query string — this is the concrete mechanism behind the CLAUDE.md's "design for keyword mode: alias-expanded OR-queries + `term_clauses`."

**Events/surfaces — this is where the June 30, 2026 platform shift (already flagged as a "known spec bug" in Curie's CLAUDE.md) gets fully confirmed.** As of the Agent messaging experience changelog: **new apps can only use `agent_view`**; the legacy Assistant path (`assistant_view`, with its `Assistant` class `threadStarted`/`threadContextChanged`/`userMessage` callbacks wrapping `assistant_thread_started`/`assistant_thread_context_changed`/`message.im`) is closed to new apps and will eventually be deprecated for existing ones. Concretely, `agent_view` changes: (1) **`app_home_opened` replaces `assistant_thread_started`** as the signal that a user opened a DM with the agent — the DM surface now looks like a normal 1:1 conversation, no separate History/Chat tabs; (2) **suggested prompts** move to the top of the Messages tab (via `assistant.threads.setSuggestedPrompts`) and `thread_ts` is no longer required; (3) after a user's new message, calling `assistant.threads.setStatus` on that thread auto-opens it to continue the conversation. Minimum versions: **Slack CLI v4.4.0, Python SDK v3.43.0 + Bolt Python v1.29.0, `@slack/web-api@7.18.0`** — these are the exact same pins already locked in Curie's CLAUDE.md, independently confirmed. [Introducing the Agent messaging experience](https://docs.slack.dev/changelog/2026/06/30/agent-messages-tab)

Beyond the DM/agent-container surface, Slack's full **entry-point taxonomy** for agents (independently confirmed against Curie's plan) is: `app_mention` (channel mentions, always reply in-thread via `thread_ts ?? event.ts`), `message.im` (DMs, filter `channel_type === 'im'`), the agent container/split-pane (gated behind "Agents & AI Apps" in app settings — this is the `assistant_view` path Curie is explicitly avoiding), **App Home** (`app_home_opened` → `views.publish`, recommended as "the persistent surface for workflow visibility... show running workflows, recent completions, blocked items... pause/resume/stop/retry/redirect actions"), modals (`views.open` for structured multi-step input), slash commands (**not supported in the split-view container or in threads generally** — only in top-level channel messages), message shortcuts (full message payload passed to the handler — "the recommended pattern for 'act on this message'" actions like "summarize"/"create a ticket"), global shortcuts (`+` compose-box button), and link unfurls (`link_shared` → `chat.unfurl`). [Interaction surfaces and entry points](https://docs.slack.dev/ai/agent-entry-and-interaction)

**Streaming** (confirmed against Curie's spec): three Web API methods — `chat.startStream` → `chat.appendStream` → `chat.stopStream` — with new Block Kit elements (`feedback_buttons`, `icon_button`, `context_actions` block) purpose-built for AI response interactivity (thumbs up/down, delete-response). Both Python and Node SDKs ship a `streamer` helper utility wrapping the three-call sequence. [New features designed for Slack apps sending AI responses](https://docs.slack.dev/changelog/2025/10/7/chat-streaming)

**Steal:** the **structured-state-object pattern is Slack's own first-party recommendation**, not a Curie invention — cite it directly to justify the pipeline architecture in `/docs/backend.md`. Also steal the **entry-point-specific reply idiom** (`thread_ts ?? event.ts`) verbatim — it's the correct, Slack-endorsed idiom for "reply in the right thread regardless of whether this is a top-level message or already in a thread."

**Avoid:** treating RTS as a general-purpose search engine substitute for a real index — it is scoped, rate-limited (≤20 results/page, action-token-gated), and explicitly not to be persisted. Any Curie feature that wants "search everything since the beginning of time" will hit real limits; the `before:`/`after:` filters and `≤3 RTS calls per verdict` budget already in Curie's CLAUDE.md are the correct mitigation, not a workaround to route around.

# IMPLICATIONS FOR CURIE

## (a) Data model

1. **Slack Lists is Curie's knowledge graph substitute — treat it with the discipline Guru and PromptQL apply, not the informality of a spreadsheet.** Glean's lesson is that raw LLM retrieval fails specifically on multi-hop, deterministic, enumerate-style questions ("has this exact reagent+cell-line+assay combination been tried") — which is *exactly* Curie's core query. Don't let the LLM eyeball a dump of List rows; query the List with real filters (List API supports filtered `.items.list`) the same way PromptQL insists on query-pushdown over prompt-stuffing. Concretely: define a versioned schema for the "past experiments" List (columns: hypothesis, method summary — as `rich_text`, reagents/cell-lines as structured multi-select or `message`-type columns for provenance-linkback, outcome enum [success/failure/inconclusive/superseded], date, `parent_item_id` for follow-up-experiment hierarchies) and treat schema changes as an explicit, reviewed step, not ad hoc.

2. **Add a verification/confidence field to every List row, à la Guru — but make it a ranking signal, not a gate.** A failed-experiment record entered in haste should still count as prior art on day one (unlike Guru's hard "unverified = agent can't use it" rule, which would make Curie useless until someone does manual review). Instead: store `confidence: {high | medium | low}` and `last_confirmed_at`, and feed both into the verdict ranking (Unblocked's authority/freshness-based tie-break: List record > recent RTS message with explicit outcome language > older RTS message > literature), with the LLM prompt instructed to hedge language ("a lab record from 3 months ago suggests...") when confidence is low rather than refusing to use the record.

3. **Hash-gate re-verdicts, à la Notion's Page State.** Store a normalized hash (reagents + method + hypothesis, stripped of formatting/whitespace) per experiment-plan message. If a scientist edits a plan message and the hash is unchanged, skip the full preflight→RTS→scholar→verdict pipeline and re-post the cached verdict — this directly protects the `≤3 RTS calls per verdict` budget and avoids burning literature-API quota on cosmetic edits.

4. **Hypothesis/belief state is a small, explicit structured object, not a prose blob** — following both Slack's own `{goal, constraints, decisions, artifacts, sources}` recommendation and Dust's Tracker pattern (staleness detection over structured signals, not vibes). Model: `{hypothesis_id, statement, prior_confidence, evidence[] (each {source_type: list_record|rts_message|literature, permalink, summary, supports|contradicts}), last_updated, superseded_by}`. Belief-update posts are diffs against this object, which is what makes "posts belief updates" a well-defined, testable feature rather than a fuzzy re-summarization.

5. **Do not build a vector database or persist Slack content.** Lean entirely on Slack's own semantic mode in RTS (`disable_semantic_search: false` for natural-language queries, `true` for exact keyword mode) and cache only what the platform's terms of service and Curie's own CLAUDE.md already permit — computed verdicts, hashes, and List-row references (which are Slack-owned data accessed live, not copied).

## (b) Event listeners

1. **`app_home_opened` (checking `tab === "messages"`) + `message.im` for the DM loop — confirmed correct, do not build the legacy `Assistant` class.** This research independently confirms Curie's CLAUDE.md smoke-test bias: `agent_view` is mandatory for new apps as of the June 30, 2026 changelog, `assistant_thread_started`/`assistant_thread_context_changed` no longer fire meaningfully for DM-open detection under `agent_view`, and the required SDK pins (Bolt Python ≥1.29.0, slack-sdk ≥3.43.0, CLI ≥4.4.0) match exactly. Treat smoke test #8 as confirmatory, not exploratory — you can skip straight to `agent_view` with high confidence.

2. **`app_mention` is the primary collision-check trigger** (scientist posts an experiment plan and @-mentions Prior, or is in a channel where Prior auto-triggers) — always reply via `thread_ts ?? event.ts` per Slack's own idiom. Extract `event.action_token` here for the RTS call (bot-token calls require it, and it's only present on `app_mention` and `message.im` — this is *why* those are the only two events Curie's RTS tool can call from).

3. **Message shortcut ("Check this against prior work")** as a secondary entry point for the "act on this message" pattern Slack explicitly recommends for summarize/ticket-create/reply-draft style actions — lets a scientist retroactively run a collision check on an experiment plan that was posted without an @-mention, without needing to repost it. Low engineering cost (reuses the same preflight→verdict pipeline) and directly extends coverage.

4. **App Home as the belief-ledger/status surface**, matching Slack's explicit recommendation ("persistent surface for workflow visibility... show running workflows, recent completions, blocked items... pause/resume/stop/retry"). Publish on `app_home_opened` with `tab === "home"`: recent verdicts, open hypotheses with confidence trend, and a retry button for any failed check — this is also where the "🤖 Curie" AI disclaimer and "compiled by Curie" branding belongs per the CLAUDE.md naming rules, since App Home is product-surface, not agent-conversation surface.

5. **`link_shared` unfurl for List-record permalinks** — low-cost, high-polish: when a scientist shares a link to a Curie-authored List entry or a permalink Curie cited in a verdict, unfurl it with a small Block Kit card (hypothesis, outcome, confidence) instead of Slack's default plain-link preview. This reinforces "the List is the source of truth" as a visible, referenceable surface rather than a black box only Prior reads.

6. **Explicitly do not implement:** `assistant_thread_started`, `assistant_thread_context_changed`, or the Bolt `Assistant` class — already correctly forbidden in CLAUDE.md; this research adds no new information that should reopen that decision.

## (c) Tools/actions

1. **`tools/record_store.py` (Slack Lists wrapper) should expose a *filtered query*, not a full-dump read** — mirror PromptQL's query-pushdown principle. Build the equivalent of `slackLists.items.list` with server-side filters where the API allows, and where it doesn't, filter client-side *before* constructing the LLM prompt (never paste the entire List into context). Writes (`slackLists.items.create/.update`) must encode `message`/`canvas`-type rich_text payloads per the CLAUDE.md's Block Kit `rich_text` requirement — validate this at the tool boundary with a pydantic model so malformed payloads fail fast in `llm/client.py`'s JSON-validation layer rather than silently corrupting a List row.

2. **`tools/rts.py` should implement the alias-expansion + `term_clauses` OR-query pattern the CLAUDE.md already specifies, with an explicit rank/dedupe step modeled on Unblocked's authority-ranking** rather than trusting RTS's default `score` sort alone. Concrete tie-break order: (1) Slack List record match (deterministic, highest authority) → (2) RTS message containing explicit outcome language ("failed," "didn't work," "abandoned" — a small keyword set, not ML) → (3) other RTS matches sorted by recency → (4) literature. Cap at ≤3 RTS calls total (one broad semantic query + up to two narrower keyword follow-ups using `term_clauses`), consistent with the CLAUDE.md's rate ceiling. Always request `include_context_messages: true` and surface `permalink` fields into the `sources` list the CLAUDE.md's frontend contract requires — this is directly specified in Slack's own docs as "a best practice."

3. **`tools/scholar.py` (literature search) should be treated as the lowest-authority, highest-latency source** — call it last, only if RTS + List checks don't already produce a confident verdict, and cache aggressively (paper metadata doesn't change hour to hour, unlike lab Slack chatter) — this both respects the calibration mandate (avoid manufacturing false collisions from a tangentially-related paper outranking a clear internal "we tried this" record) and keeps the pipeline's latency budget sane for Slack's response-time expectations.

4. **A `tools/ledger.py` staleness-detection tool, modeled directly on Dust's Tracker**, should run as a background/scheduled check (not inline with a user's message) that scans open hypotheses in the belief-state store for ones referencing List records or RTS evidence that has since been superseded or contradicted by a newer record — then posts a belief-update proactively rather than waiting for a scientist to ask. This is the concrete mechanism behind "posts belief updates" in the product brief, and Dust's field validation (a live product doing exactly this against Notion/Confluence docs) is good evidence the pattern works in production, not just in theory.

5. **Every tool call should append to a structured trace object** (tool name, query, result count/top permalink, latency, token cost), modeled on BubbleLab's execution-trace logging — surfaced as the "Checking priors…" stream `task_update` chunks with sources (per the CLAUDE.md's streaming contract) *and* persisted (verdict + trace, never raw Slack content) so a scientist who disputes a verdict can see exactly which List rows and messages drove it. This is what makes "false collision is the unforgivable bug" auditable rather than just aspirational.

6. **Do not build:** a custom embedding pipeline, a persisted vector index, or a general entity-extraction/knowledge-graph builder. Every product surveyed that built one (Glean, Notion, Dust's connector-ingestion layer) did so as a multi-year, dedicated-infra investment; Curie's entire "knowledge graph" should be Slack Lists' native rows/columns/hierarchy plus Slack's own RTS semantic mode, with zero custom ML infrastructure — this is both the pragmatic hackathon-deadline choice and, per Slack's terms of service already cited in the CLAUDE.md, the compliant one.
